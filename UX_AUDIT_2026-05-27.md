### `app/app/cases/page.tsx` (~640 lines)

**[MOBILE]** **Kanban grid snap-scroll UX is undiscoverable**  
`line 522–525`: The grid renders all columns at `minmax(260px, 1fr)` with `overflow-x-auto snap-x`. On mobile the user sees one column and must swipe horizontally. The pill switcher at `line 503` clicks into view via `scrollIntoView` — but this interaction is not communicated anywhere (no swipe hint, no "← →" indicator). Users will think the board is a single column.  
*Fix:* Add a swipe hint `"← swipe to see more stages"` below the column switcher, or implement a true single-column view with a `select` dropdown for the active stage.

**[MOBILE]** **Modal close button is 28×28px** (`line 178–181`)  
```tsx
className="w-7 h-7 rounded-md hover:bg-white/8 ..."
```
`w-7 h-7` = 28×28px. iOS requires 44×44px minimum tap target. The entire Create Case Modal close button is unreachable for many users.  
*Fix:* Change to `w-11 h-11` (44px) or add `p-2` padding: `p-2 -m-2`.

**[MOBILE]** **Tab strip buttons are 36px tall** (`line 488`)  
`h-9` = 36px. Under the 44px threshold. Tabs are the primary navigation element on this page.  
*Fix:* Change to `h-11`.

**[UX]** **Drag-and-drop does not work on touch devices** (`lines 403–418, 563–574`)  
The kanban uses `draggable`, `onDragStart`, `onDragOver`, `onDrop` — native HTML5 drag API. This API does **not** fire on iOS Safari or most Android browsers. Cards cannot be moved on mobile.  
*Fix:* Use `@dnd-kit/core` (already popular in React projects) or add a `onTouchStart`/`onTouchEnd` fallback that opens a stage-select dropdown.

**[UX]** **`refreshCases` duplicates 30 lines of mapping logic** (`lines 420–445` vs `lines 294–323`)  
The case-mapping code is copy-pasted verbatim into the refresh function. Any bug fix must be applied twice.  
*Fix:* Extract `mapCasesToCards(items: any[]): Card[]` as a module-level function.

**[A11Y]** **Modal close button has no `aria-label`** (`line 178`)  
`<button onClick={onClose}><X size={14} /></button>` — screen readers announce "button" with no label.  
*Fix:* Add `aria-label="Close"`.

**[A11Y]** **Mobile column switcher pills have no ARIA role** (`lines 503–519`)  
These pill buttons toggle the visible kanban column — functionally they are tabs. They have no `role="tab"`, no `aria-selected`, and no `aria-controls`.  
*Fix:* Add `role="tab"` + `aria-selected={mobileCol === col.id}` + `aria-controls={`col-${col.id}`}`.

**[VISUAL]** **Dead CSS string concatenation** (`line 545`)  
```tsx
className={`snap-start rounded-2xl border p-3 transition ${ "" } ${...}`}
```
A template literal with an empty string `${ "" }` suggests removed conditional styling left behind.  
*Fix:* Remove the empty interpolation.

**[PERF]** **Four independent `useEffect` fetches fire simultaneously on mount** (`lines 294, 326, 343, 360`)  
Cases, requests, resources, and reports each trigger a separate API call on mount. No dependency coordination. On a slow connection this creates 4 parallel edge function invocations.  
*Fix:* Combine into `Promise.all([...])` in a single `useEffect`, or lazy-load non-active tabs.

### `app/app/match/page.tsx` (~740 lines)

**[MOBILE]** **Board mode forces `min-w-[1100px]`** (`line 138`)  
```tsx
<div className="grid gap-3 min-w-[1100px]" ...>
```
Unlike the cases kanban which has a mobile column switcher, the match board has no mobile fallback. On a 375px screen, the board is 1100px wide with horizontal scroll and **no column switcher**. Users can scroll to see all 5 columns but there's no guidance and no way to focus a single column.  
*Fix:* Add the same mobile column-switcher pill pattern from `cases/page.tsx:503`.

**[MOBILE]** **Accept/Decline buttons are 36px tall** (`lines 556, 560`)  
```tsx
className="h-9 px-3 rounded-lg ..."
```
Primary action buttons (Accept a match, Decline) are `h-9` = 36px. On mobile these are the most important interactive elements on the page.  
*Fix:* Change to `h-11` (44px) or `h-10 py-2`.

**[MOBILE]** **Confirm dialog is inline, not focused** (`lines 587–597`)  
The confirmation UI for accepting a match renders inline within the card. On mobile, this pushes content down and may scroll out of view before the user notices the confirmation state.  
*Fix:* Show confirmation as a bottom sheet or modal on mobile.

**[UX]** **No error state when match API fails** (`lines 90–102`)  
The `MatchBoard` component catches errors with `console.error` only. If the API call fails, `matches` stays `[]` and the board shows empty columns with no explanation.  
*Fix:* Add an `error` state that renders a "Failed to load matches — retry" card.

**[UX]** **`MatchWorkbench` (match mode) shows no loading state for candidate fetch**  
The workbench has `loading` for the case list, but when a case is selected and candidates are fetched, there is no indicator — the right panel just stays blank until results arrive.  
*Fix:* Add a spinner or skeleton to the candidates panel during fetch.

**[A11Y]** **Score bars have no text alternative** (`ScoreBar` component)  
Match score bars are visual gradients with no text. A screen reader user cannot determine which candidate scores higher.  
*Fix:* Add `aria-label={`Score: ${score} out of 100`}` to score bar containers.

**[A11Y]** **Color-only urgency encoding**  
Urgency is encoded via dot color (`um.dot`) only. No text label for the urgency level in the match board cards (`line 182`).  
*Fix:* Add urgency text next to the dot: `<span className="sr-only">{urgency}</span>`.

**[VISUAL]** **Chain view uses `grid-cols-[1fr_auto_1fr_auto_1fr]` with no mobile breakpoint** (`line 632`)  
```tsx
className="grid grid-cols-1 md:grid-cols-[1fr_auto_1fr_auto_1fr] ..."
```
This correctly uses `md:` — the three-way chain card stacks vertically on mobile. ✓ Good. However the `ChainArrow` component still renders between stacked cards and looks like a downward chevron that could be confused for a collapse toggle.

**[A11Y]** **Capacity/condition indicators are color-only**  
Facility capacity bars (if rendered) encode occupancy via fill color alone.

### `app/app/transport/page.tsx` (~478 lines)

**[MOBILE]** **`window.prompt()` for issue reporting** (`line 122–131`)  
```tsx
const description = window.prompt("Describe the issue:");
```
`window.prompt()` is:
- Blocked by default in many browsers when in iframes or cross-origin contexts
- Not styled — shows a native OS dialog that breaks the dark theme
- Completely inaccessible (no ARIA, no keyboard trap management)
- Deprecated UX pattern for PWAs and mobile web  
*Fix:* Replace with an inline form or a modal/bottom sheet: `setShowIssueFormId(assignmentId)`.

**[MOBILE]** **Input fields lack mobile-optimized `type` attributes** (`lines 176–200`)  
Driver name input has `type="text"` — fine. Origin/destination inputs also `type="text"` — correct. However there is no `autocomplete` attribute to help mobile keyboards suggest addresses.  
*Fix:* Add `autoComplete="street-address"` to origin/destination inputs.

**[UX]** **Transport list fetch swallows errors silently** (`line 73, 80`)  
```tsx
.catch(() => {})
```
If `api.transportList` fails, `assignments` stays `[]`, `loading` becomes `false`, and the page shows an empty list. No toast, no error message.  
*Fix:* ```tsx
.catch(() => toast.error("Failed to load assignments"))
```

**[UX]** **KPI counts are `useMemo` over `assignments`, but `assignments` starts as `[]`**  
During loading, KPIs display "0 Active, 0 In Transit, 0 Delivered" — which looks like real data, not a loading state. Users may conclude there are genuinely no active transports.  
*Fix:* Show `—` or a skeleton while `loading === true`.

**[VISUAL]** **List/Map toggle buttons are 28px tall** (`line 149–154`)  
`h-7` = 28px for the List/Map toggle buttons. Below minimum touch target.  
*Fix:* Change to `h-9` or `h-10`.

### `app/app/calendar/page.tsx` (~531 lines)

**[UX]** **Calendar week is hardcoded to March 16–22** (`lines 27–35`)  
```tsx
const days = [
  { label: "Mon", date: "Mar 16" },
  { label: "Tue", date: "Mar 17" },
  ...
];
```
Today is 2026-05-27. The calendar permanently shows the wrong week. Any events created will display on "Mar 16–22" labels regardless of their actual date.  
*Fix:* Compute the current week dynamically:
```tsx
const days = Array.from({ length: 7 }, (_, i) => {
  const d = new Date();
  d.setDate(d.getDate() - d.getDay() + 1 + i);
  return { label: d.toLocaleDateString('en', { weekday: 'short' }), date: d.toLocaleDateString('en', { month: 'short', day: 'numeric' }) };
});
```

**[UX]** **API error silently falls back to empty state** (`lines 53–55`)  
```tsx
.catch(() => {
  // fallback: keep seedEvents
});
```
The comment mentions `seedEvents` but that variable doesn't exist. If the API fails, the calendar shows nothing. No error message.  
*Fix:* Show a toast: `.catch(() => toast.error("Failed to load events"))`.

**[UX]** **No loading state during event creation** (`lines 63–90`)  
`addEvent` performs an API call but the form closes immediately (optimistic). If the API call fails, a rollback should occur — but there is no rollback logic. The optimistic ID remains in the list.  
*Fix:* On API failure, remove the optimistic entry: `setItems(prev => prev.filter(e => e.id !== optimisticId))`.

**[MOBILE]** **7-column week grid overflows on mobile**  
A `grid-cols-7` layout for the week view would be ~53px per column on a 375px screen — too narrow for event text. (Implementation not fully read, but the `days` array drives a 7-column layout.)  
*Fix:* On mobile, show a single day column with a date-picker strip to navigate days.

**[A11Y]** **Calendar grid is likely not semantically a grid**  
If the days use `<div>` grid layout rather than `<table role="grid">`, screen reader navigation (`←→↑↓` arrow keys) won't work.  
*Fix:* Use `role="grid"` + `role="row"` + `role="gridcell"` or switch to a `<table>`.

### `app/app/volunteers/page.tsx` (~477 lines)

**[UX]** **API errors silently swallowed** (`line 70`)  
```tsx
.catch(() => {});
```
If `api.crmVolunteersAvailable` fails, the page shows an empty list with no message.  
*Fix:* `.catch(() => toast.error("Failed to load volunteers"))`.

**[UX]** **Volunteer type mismatch between state type and mapped type**  
`useState<typeof volunteerDetails[number][]>([])` types volunteers as `VolunteerDetail` (with `towCapacity`, `availability`, `deployments`, `credentials`), but `mapVolunteers()` returns a simpler shape (`{id, name, skills, hours, status}`). The detail drawer accesses `vol.credentials`, `vol.deployments`, etc. — all `undefined` on live data. No null guard.  
*Fix:* Either align types, or guard: `vol.credentials ?? []`, `vol.deployments ?? []`.

**[MOBILE]** **Add volunteer form inputs lack `type` attributes**  
Email field (`addEmail`) has `type="text"` instead of `type="email"`. Phone field (`addPhone`) has `type="text"` instead of `type="tel"`. Mobile keyboards won't switch to email/numeric mode.  
*Fix:* Add appropriate `type` attributes to the email and phone inputs.

**[A11Y]** **Skill filter pills have no selected state communicated to assistive tech**  
Skill filter buttons change visual styling when active but have no `aria-pressed` attribute.  
*Fix:* Add `aria-pressed={skillFilter === skill}`.

### `app/app/command/page.tsx` (~100 lines)

**[UX]** **Hardcoded `DEMO_ORG_ID` is a production ERV org UUID** (`line 11, 23`)  
```tsx
const DEMO_ORG_ID = "9ad0f2ad-7789-47a8-bfba-0ae3382c86cc";
const orgId = authOrgId || DEMO_ORG_ID;
```
If auth context fails to load (race condition, session expiry), this page silently falls back to querying real ERV production data as an unauthenticated user. This is a security concern and a UX deception — the page appears to work when it shouldn't.  
*Fix:* Remove the `DEMO_ORG_ID` fallback. If `authOrgId` is null, show a loading state or redirect to login.

**[UX]** **StatPill shows `undefined` during load** (`line 55`)  
`value={stats?.openRequests}` — while `stats` is `null`, the pill renders `undefined`. This shows as blank text rather than `—` or a spinner.  
*Fix:* `value={stats?.openRequests ?? "—"}` (already the prop type allows this via the `| undefined | null` union — just needs the fallback in the render).

**[MOBILE]** **Stats strip scrolls horizontally but has no scroll indicator** (`line 54`)  
`overflow-x-auto scrollbar-hide` hides the scrollbar. On mobile, users won't know there are more stats to the right (Resources, People).  
*Fix:* Add a right-edge gradient fade as a scroll affordance: `after:absolute after:right-0 after:inset-y-0 after:w-8 after:bg-gradient-to-l after:from-[var(--surface-app)]`.

### `app/app/map/page.tsx` + `page.production.tsx`

**[MOBILE]** **No mobile-specific Mapbox controls**  
Mapbox GL JS requires explicit configuration for touch zoom and rotation. Without `dragRotate: false` (or a clear gesture guide), two-finger rotate on mobile can confuse users expecting two-finger zoom.  
*Fix:* Add `map.dragRotate.disable()` and `map.touchZoomRotate.disableRotation()` for mobile, or add a compass reset button.

**[MOBILE]** **Map takes `calc(100vh - 56px)` height**  
On mobile browsers with dynamic address bar (iOS Safari), `100vh` is the *maximum* viewport height including the hidden address bar. Content at the bottom of the map will be obscured by the address bar when it reappears.  
*Fix:* Use `100dvh` (dynamic viewport height) instead of `100vh`.

**[A11Y]** **Map has no keyboard-navigable alternative**  
Users who cannot use a touch screen or mouse cannot interact with the map at all. No list view of map markers is provided.

## 4. Quick Wins (< 5 minutes each)

Each of these can be fixed with a one-line change:

1. **Fix calendar hardcoded dates** — replace the `days` constant with `Array.from({length:7}, ...)` computed from `new Date()`.

2. **Add error toast to transport fetch** — change `catch(() => {})` to `catch(() => toast.error("Failed to load assignments"))` at `transport/page.tsx:73,80`.

3. **Add error toast to volunteers fetch** — same pattern, `volunteers/page.tsx:70`.

4. **Remove DEMO_ORG_ID fallback** — change `authOrgId || DEMO_ORG_ID` → `authOrgId` at `command/page.tsx:23`. If null, show loading state.

5. **Fix StatPill undefined** — add `?? "—"` to all five `value=` props at `command/page.tsx:55–59`.

6. **Fix modal close button touch target** — change `w-7 h-7` → `w-11 h-11` on the Create Case modal close at `cases/page.tsx:179`.

7. **Fix Map viewport height** — change `100vh` → `100dvh` in map layout/page.

8. **Add `aria-label="Close"` to all icon-only close buttons** — cases modal (`line 178`), transport form, volunteer drawer. Grep `<X size=` to find them all.

9. **Fix Reports loading state** — add `const [loading, setLoading] = useState(true)` and `.finally(() => setLoading(false))` in `reports/page.tsx`.

## 5. Systemic Recommendations

### Error Handling Standardization
Every page has a unique approach to error handling. Create a shared pattern:
```tsx
// lib/use-api-fetch.ts
function useApiFetch<T>(fetcher: () => Promise<T>, errorMsg: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(false);
  useEffect(() => {
    fetcher()
      .then(setData)
      .catch(() => { setError(true); toast.error(errorMsg); })
      .finally(() => setLoading(false));
  }, []);
  return { data, loading, error };
}
```

### Color Token Consolidation
Colors are hardcoded across 50+ files. Consider adding to `tailwind.config.js`:
```js
colors: {
  sos: { red: '#EF4E4B', blue: '#89CFF0', green: '#34D399', cream: '#F5EBD6' }
}
```
Then grep-replace `#EF4E4B` → `sos-red`, etc.

### Touch Target Audit Script
To find all sub-44px buttons, run:

grep -rn "w-[67]\b\|w-8\b\|h-[678]\b\|h-9\b" app/app/ --include="*.tsx"
```
This will surface all candidates for touch target fixes.

### `window.prompt()` Elimination
Search for all usages:

grep -rn "window\.prompt\|window\.confirm\|window\.alert" app/ --include="*.tsx"
```
Replace every instance with a proper modal or inline form.
